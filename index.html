Quick Rules & Explanation

How encoding works

  1. Convert your plaintext to a numeric string using the chart above.
  2. The first 5 digits of your pad are the pad identifier — they are copied directly to the front of the encoded message unchanged.
  3. For every subsequent plaintext digit, subtract the corresponding pad digit and apply mod 10:
    encoded = (plaintext − pad + 10) mod 10
  4. The final encoded message = pad identifier (5 digits) + all encrypted digits.

How decoding works

  1. The first 5 digits of the encoded message and the first 5 digits of the pad must match — that is the pad identifier.
  2. For every subsequent encoded digit, add the corresponding pad digit and apply mod 10:
    plaintext = (encoded + pad) mod 10
  3. Convert the resulting numeric string back to text using the chart.

Letter vs. Number mode

  • The decoder always starts in Letter mode.
  • The code 90 (FIG) toggles between Letter mode and Number mode.
  • In Letter mode, single-digit codes 1–6 map to A E I N O T; two-digit codes 70–99 map to the rest of the alphabet and special characters.
  • In Number mode, digits 0–9 decode directly as numeric characters.
  • To encode a run of digits in a message, a FIG (90) is inserted before the run and another FIG (90) is inserted after.

Classroom security note

This app uses the browser's crypto.getRandomValues() API, which is cryptographically strong for classroom demonstrations. However, a true one-time pad requires:

  • Truly random pads (from physical entropy sources)
  • Strict key secrecy and one-time-only use
  • Secure out-of-band pad exchange between parties
  • Immediate destruction of the pad after use

Browser-generated pads are not suitable for real cryptographic security.
